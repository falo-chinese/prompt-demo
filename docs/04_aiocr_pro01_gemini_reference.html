FALO Prompt Manager Reference

AI_Teach_Classroom / aiocr-pro01 Gemini OCR 參考整理

把 falo-taiwan/aiocr-pro01 與本機 AI_Teach_Classroom 的 Gemini OCR 模型路由、影像處理策略與資料邊界，整理成 Prompt Manager v0.3 的參考設計。

1. Pro01 的核心定位

aiocr-pro01 不是單純發票 OCR 頁面，而是一個 local-first 的文件 AI ETL / HITL workbench。

image / document / form
-> template field schema
-> prompt schema
-> AI1 / AI2 proposal
-> human reconciliation
-> confirmed result
-> evidence package export
對 Prompt Manager 的啟發：OCR 後要進入 Prompt、Schema、人工檢查、JSON 匯出；v0.3 只需要保留資料入口與題庫，不先做完整 reconciliation table。

2. Gemini 模型路由參考

這些模型名稱來自參考專案與本機文件；未來真正接 API 前，要再確認當下官方可用模型。v0.3 只把它放成「模型菜」裡的廚師選項，不直接啟動 API。

4. 影像處理策略參考

影像準備和 prompt 一樣重要。教材上要讓學員理解，OCR 不是「越大越好」，而是要清楚、可讀、成本合理、可追蹤。

關鍵原則：不要放大小圖。放大不會創造真實細節，只會增加傳輸與模型成本。

5. Metadata 與資料保存邊界

  • API key 不寫入公開原始碼；v0.3 預留欄位不保存 key。
  • 若未來教學 demo 暫存 key，必須有清楚的清除按鈕與儲存說明。
  • 本地 DB、JSON export、Excel export 不保存原始圖片 bytes。
  • 工作畫面可以顯示當前圖片，但持久化紀錄只保留 metadata 與分析結果。

6. Multi-source Vision Runtime 觀念

OCR / Vision Source Adapter
-> Prompt-based Extraction
-> Structured JSON
-> Normalization
-> Audit Trail
-> HITL
-> ERP / Workflow ETL

Gemini Vision、AI Browser、PaddleOCR、Human Input 都可以是資料來源。Prompt Manager v0.3 不處理 runtime orchestration，但要先把 Prompt、Schema、Workflow 位置設計對。

7. Prompt Manager v0.3 採用方式

  1. OCR / Gemini 預留區文案註明 AI_Teach_Classroom 與 Pro01 參考策略。
  2. OCR Prompt Pack 補一張「Gemini OCR 影像策略檢查」卡片。
  3. 文件記錄模型路由、影像策略、metadata 保存邊界。
  4. JSON Connect 繼續作為主線，不強迫本系統直接呼叫 Gemini API。

v0.3 不做完整圖片上傳與 Gemini API OCR、不做 AI1 / AI2 reconciliation table、不做圖像 base64 持久化。

8. 教學說法

OCR 是把紙本食材洗乾淨；影像策略是切菜大小；Gemini 模型是廚師；JSON Connect 是把成品裝成標準餐盒；人工審查是出餐前品管。
